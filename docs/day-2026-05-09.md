## 3. 关键设计决策（面试重点）

1. 为什么 repository 分 interface 和 mysql impl？
- service 依赖抽象，降低耦合。
- 易测试（mock/fake repository）。
- 易替换（MySQL -> PG/其他存储）。

2. 为什么要拆 StatementRepository 与 StatementQueryRepository？
- 读写职责分离（轻量 CQRS 思路）。
- 复杂查询不会污染写接口。
- 便于后续针对查询单独优化（缓存/读库/SQL）。

3. 为什么 app.go 还要“注册很多模块”？
- 启动时集中装配（composition root）是显式依赖管理。
- 通过 `bootstrap/modules` 把复杂度下沉，`app.go` 只做 orchestration。

## 5. 面试官高频追问（Q/A）

### Q1: 你们为什么不用 ActiveRecord 风格把所有逻辑写在 model？
A: Go 更强调显式边界。把 HTTP、业务、数据访问拆开后，测试、替换和维护成本更低，复杂查询也不会污染业务对象。

### Q2: 隐式接口实现有什么价值？
A: service 只依赖接口，不关心具体 mysql 实现；同一实现可以同时满足多个接口，便于组合和替换。

### Q3: 你如何避免 `order_by` SQL 注入？
A: 采用白名单映射，不直接拼接用户输入。

### Q4: current_user 是如何传递的？
A: API 鉴权中间件校验成功后 `c.Set("current_user", user)`；handler 使用统一 helper 获取。

### Q5: 为什么 `check_openid` 不走鉴权中间件？
A: 它本身是拿 session 的入口，不能要求已有 session。
